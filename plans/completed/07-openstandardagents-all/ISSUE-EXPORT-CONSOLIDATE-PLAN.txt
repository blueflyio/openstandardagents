## 1. EXPORT ALL GITLAB ISSUES (LOCAL)

### 1.1 Prerequisites
- Token: GITLAB_TOKEN or GITLAB_TOKEN_PAT in /Volumes/AgentPlatform/.env.local (or platform config).
- BuildKit: From agent-buildkit worktree, run:
  set -a && [ -f /Volumes/AgentPlatform/.env.local ] && source /Volumes/AgentPlatform/.env.local; set +a
  cd $HOME/Sites/blueflyio/worktrees/agent-buildkit

### 1.2 Export commands (run in order)

# Open issues only (default)
node dist/cli/index.js gitlab issues list --group blueflyio --state opened --max 0 --json > tmp/issues-export-opened.json

# All issues (opened + closed) for full snapshot
node dist/cli/index.js gitlab issues list --group blueflyio --state all --max 0 --json > tmp/issues-export-all.json

# Optional: count only (sanity check)
node dist/cli/index.js gitlab issues list --group blueflyio --state opened --count-only

Do not pipe through head -c; that truncates JSON and breaks parsing.

### 1.3 If token check fails
Run: node dist/cli/index.js gitlab token check
If 401/invalid: set GITLAB_TOKEN or GITLAB_TOKEN_PAT in /Volumes/AgentPlatform/.env.local (create PAT at https://gitlab.com/-/user_settings/personal_access_tokens, scopes: api, read_repository, write_repository).

## 2. DIVIDE BY PROJECT

### 2.1 Parse export
Group issues by project. Each issue from the group API has:
- iid, title, state, web_url, project_id (optional), references.relative (e.g. "group/project#123").

Extract project path from web_url (e.g. https://gitlab.com/blueflyio/agent-platform/tools/agent-buildkit/-/issues/123 -> blueflyio/agent-platform/tools/agent-buildkit) or from references.relative.

### 2.2 One-time script (optional)
In workspace tmp/, a small Node script can:
- Read tmp/issues-export-all.json
- Group by project path (parse web_url or references)
- Write tmp/issues-by-project.json { "project/path": [ issue, ... ], ... }
- Print per-project counts and list of projects with open count.

No new .sh; use Node/ts-node or buildkit if you add a subcommand later.

### 2.3 Manual alternative
Open issues-export-opened.json in an editor; search for "web_url" and group by the path segment (blueflyio/.../project-name). Or use jq:
jq 'group_by(.web_url | split("/")[4:6] | join("/")) | map({project: .[0].web_url | split("/")[4:6] | join("/"), count: length, issues: map(.iid)})' tmp/issues-export-opened.json

## 4. CLOSE EXISTING ISSUES & BUILD NEW

### 4.1 Rules (mandatory)
- Do NOT close an issue (or epic) unless the code is merged to the release branch and into main (per AGENTS.md).
- Orphan issues: close only if duplicate, obsolete, or consolidated into another issue (comment with "Consolidated into #X" before closing).

### 4.2 Closure strategy
1. Export and divide (steps 1-2). Produce:
   - List of projects with open-issue count.
   - List of issues with no epic (--no-epic) as candidates for linking or consolidation.
2. Triage (use n8n GitLab nodes or GitLab Duo; no new BuildKit bulk commands per AGENTS.md):
   - Link every orphan issue to an Epic or parent, or mark "to close: duplicate/obsolete" with a comment.
   - Do not close in bulk; close only after human or script confirms each.
3. Consolidation:
   - For each project (or group), pick a small set of "umbrella" issues or epics (e.g. "Release v0.1.x", "OSSA Fleet", "Drupal Phase 8").
   - Add comments to micro-issues: "Superseded by #X" or "Consolidated into Epic Y" then close.
   - Prefer updating existing issues over creating new ones.

### 4.3 Build new (after cleanup)
- New issues: create only when linked to an Epic or parent. Use buildkit gitlab issues create --project <path> and link via GitLab UI or API (issue link type "relates to" / "parent").
- New epics: create at group level for big themes (e.g. "Platform health", "OSSA 0.4", "Drupal CMS 2.0"). Then create few child issues per epic, not hundreds of micro-issues.
- Todo-derived work: create one issue per coherent chunk (e.g. "Complete OSSA MASTER-PLAN Phase 0" with checklist in description), link to epic, then close or archive the local todo file.
