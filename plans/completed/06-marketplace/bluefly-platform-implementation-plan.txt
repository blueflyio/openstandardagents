1. Categorized Change Audit
This section audits all recent architectural changes, API contracts, agent lifecycle, execution model, and security posture against the canonical platform state defined in AGENTS.md, CLAUDE.md, and the AgentSkills as a Service architecture recommendation.

1.1 Architecture Changes
1.1.1 Package-First Consumption (2026-02-27)
Change: All 20 TypeScript platform services consumed via npm from GitLab registry, not git clones. Old NAS clones archived to _archive/services-agent-platform-git-clones/.
Impact: Eliminates clone drift between NAS/Oracle/Mac. Simplifies onboarding. Forces clean dependency boundaries.
Status: COMPLETE — enforced in CI; NAS clones removed.

1.1.2 Three-Tier OSSA Architecture
Change: Canonical three-tier split — Tier 1: platform-agents (YAML manifests only), Tier 2: common_npm (18 @bluefly/* TypeScript packages), Tier 3: gitlab-agent_ossa (Go).
Status: STABLE — 67 agents registered across 12 domains. CI validates manifest schema.
Gap: No automated drift detection between manifest declared capabilities and actual package exports. Proposed @bluefly/skill-registry would close this.

1.1.3 Separation of Duties Enforcement
Change: CI-enforced SoD via separation-of-duties-check from gitlab_components. compliance-engine PROJECT_BOUNDARIES and agent-mesh separation-of-duties.ts validate at runtime.
Status: ENFORCED — CI blocks cross-boundary imports.
Gap: alternative_services Phase 2 incomplete — DdevAddon entity, controllers, and forms remain in core module instead of alternative_ddev submodule. Requires migration to satisfy SoD split.

1.1.4 OSSA Four-Repo SoD
Change: Clear ownership split: openstandardagents (spec/CLI only), openstandardagents.org (marketing + thin YAML editor), openstandard-ui (the app: creator/save/pipelines), openstandard-generated-agents (artifact store only).
Status: DEFINED — boundaries documented but openstandard-ui overlap with .org editor not fully tested in CI.

1.3 Agent Lifecycle Changes
1.3.1 Fleet 7-Step Pipeline
DESCRIBE→GENERATE→CLASSIFY→GATE→ROLLOUT→VERIFY→PULSE
Status: Steps 1-3 (describe, generate, classify) functional via BuildKit CLI. Steps 4-7 require Cedar policy gates and fleet deploy infrastructure.
Gap: Cedar policy gates not enforced at runtime — tier_3 (deploy) and tier_4 (fleet-wide) gates are defined but the enforcement middleware in agent-router has no tests. ROLLOUT step depends on kagent deploy endpoint which is not implemented.

1.3.2 Drupal Agent Marketplace
Status: POC functional — Drupal backend (ai_agents_marketplace, recipe_agent_marketplace), Node frontend (NODE-AgentMarketplace).
	•	kagent deploy endpoint: NOT IMPLEMENTED — blocks one-click agent deployment from marketplace UI.
	•	User auth E2E: NOT TESTED — per-user GitLab project flow has no integration tests.
	•	Wizard E2E: No end-to-end flow covering full wizard → save → deploy → verify chain.

1.3.3 Skill Lifecycle (AgentSkills Integration)
Decision: AgentSkills integration module placed in alternative_services/alternative_skills (per AGENTS.md line 384).
Status: NOT BUILT — module directory does not exist. @bluefly/skill-registry npm package also not created. No CRUD operations for skill discovery, install, validate, publish.

1.4 Execution Model Changes
1.4.1 Agentic Flows Contract
Architecture: Drupal (ECA, FlowDrop, orchestration) ↔ TypeScript (agentic-flows definitions, workflow-engine execution) via HTTP/Tool.
Status: TypeScript side: agentic-flows has flow definitions, workflow-engine has execution runtime with 1 of 3 endpoints. Drupal side: ECA and FlowDrop functional, orchestration ServicesProviderInterface done.
Gap: 'Invoke platform flow' ECA action — the Drupal ECA action that calls workflow-engine /trigger does not exist. This is the critical bridge action that enables Drupal-initiated agentic workflows.

1.4.2 OSSA Export → n8n/Zapier Pipeline
7-step flow: manifest → parse → orchestration adapter → tool plugin → webhook → n8n node → external automation.
Status: Steps 1-4 documented and partially implemented (orchestration ServicesProviderInterface). Steps 5-7 (webhook delivery, n8n custom node, Zapier integration) require workflow-engine /trigger endpoint.

1.5 Security Changes
1.5.1 1Password Token Management
Policy: Three sinks only — BuildKit/NAS (.env.local), Oracle (env vars), Mac dev (env). No master token in files. Automation uses GitLab service-account tokens.
Status: ENFORCED — buildkit gitlab token pat-self-rotate automates PAT rotation.
Gap: No automated alerting when tokens approach expiry. n8n admin and MCP API tokens (1Password vault) lack rotation schedule.

1.5.2 Cedar Policy Gates
Status: Policy definitions exist for tier_1 through tier_4. compliance-engine validates.
Gap: No integration tests covering Cedar deny scenarios. Policy update workflow (who can modify Cedar policies, how they deploy) is undocumented.

3. Prioritized Remediation & Enhancement Plan
Sequenced into four phases with explicit dependencies. Each work item references the gap ID from Section 2.

3.1 Phase 1 — Critical Path (Weeks 1-2)
Goal: Unblock the Drupal ↔ TypeScript integration bridge and marketplace deploy.

3.1.1 Implement workflow-engine execute-by-name [G1]
	•	Project: common_npm/workflow-engine
	•	Action: Add POST /api/v1/flows/execute-by-name endpoint. Accept { name, version?, input } body. Resolve flow by name from agentic-flows registry. Return execution result.
	•	Tests: Unit tests for name resolution, integration test with agentic-flows mock, error handling for missing/ambiguous names.
	•	Deploy: Oracle via CI on merge to release/v0.1.x.

3.1.2 Implement workflow-engine /trigger [G2]
	•	Project: common_npm/workflow-engine
	•	Action: Add POST /api/v1/trigger endpoint. Accept Drupal-shaped payload (entity_type, bundle, action, data). Map to flow execution via name convention or explicit mapping table.
	•	Tests: Integration test with Drupal orchestration webhook payload. Callback verification to Drupal webhook endpoint.
	•	Dependency: Implements after 3.1.1 (shares flow resolution logic).

3.1.3 Build ECA action 'Invoke platform flow' [G3]
	•	Project: all_drupal_custom/ai_agents_ossa
	•	Action: Create ECA action plugin that calls workflow-engine /trigger. Configure via ECA UI: flow name, payload mapping, callback URL.
	•	Tests: PHPUnit for action plugin. DDEV integration test: ECA event → action → HTTP call → verify trigger received.
	•	Dependency: Requires G2 deployed on Oracle.

3.1.4 Implement kagent deploy endpoint [G6]
	•	Project: NODE-AgentMarketplace
	•	Action: Add POST /api/agents/{agentId}/deploy endpoint. Trigger kagent-based deployment pipeline. Return deployment status with polling URL.
	•	Tests: Mock kagent integration, verify pipeline trigger, test rollback on failure.

3.2 Phase 2 — Skill Infrastructure (Weeks 3-4)
Goal: Build the skill lifecycle layer and close the AgentSkills format gap.

3.2.1 Create @bluefly/skill-registry [G4]
	•	Project: common_npm (new package)
	•	Action: Full CRUD for skill manifests. GKG integration for semantic indexing. Skill validation (schema check, dependency resolution, security scan stub). Expose via MCP tool for agent-protocol.
	•	Interface: SkillRegistry { create, read, update, delete, search, validate, publish }. Zod schemas for all inputs.
	•	Tests: Unit tests for CRUD, integration test with GKG mock, MCP tool integration test.

3.2.2 Build Format Normalizer [G5]
	•	Project: common_npm (new package or module in skill-registry)
	•	Action: Bidirectional conversion: OSSA YAML manifest ↔ AgentSkills.io SKILL.md (frontmatter + markdown). Handle capability mapping, input/output schema translation, metadata preservation.
	•	Tests: Round-trip tests: OSSA → AgentSkills → OSSA with diff verification. Edge cases: multi-tool agents, nested schemas.
	•	Dependency: Requires G4 for registry integration.

3.2.3 Complete alternative_services Phase 2 [G7]
	•	Project: all_drupal_custom/alternative_services
	•	Action: Migrate DdevAddon entity type, controllers, forms, and routes from core module to alternative_ddev submodule. Update service references and dependency injection.
	•	Tests: Verify DDEV addon CRUD after migration. Run existing test suite to catch regressions.

3.2.4 Cedar policy integration tests [G8]
	•	Project: common_npm/compliance-engine
	•	Action: Write integration tests covering: tier_3 deploy gate (approve/deny scenarios), tier_4 fleet-wide gate, policy update rejection for unauthorized agents, conflict matrix enforcement.
	•	Tests: Minimum 8 test cases covering all 4 tiers and 2 deny scenarios per tier.

3.3 Phase 3 — Marketplace & Fleet (Weeks 5-6)
Goal: E2E marketplace flows and fleet skill deployment.

3.3.1 User auth E2E tests [G10]
	•	Project: NODE-AgentMarketplace
	•	Action: Integration test: user signup → GitLab project creation → agent fork → deploy → verify running. Cover OAuth flow, project permissions, cleanup on failure.

3.3.2 Wizard E2E test flow [G9]
	•	Project: openstandard-ui
	•	Action: Playwright E2E: open wizard → configure agent → save manifest → trigger pipeline → verify artifact in openstandard-generated-agents. Test abort/resume.
	•	Dependency: Requires G6 (deploy endpoint) and G10 (user auth) from Phase 1-2.

3.3.3 Build Fleet Skill Deploy [G12]
	•	Project: new component (fleet-skill-deploy in common_npm or agent-buildkit command)
	•	Action: Integrate with fleet-ops 7-step pipeline. Steps: validate skill (schema + security) → classify target fleet → Cedar gate check → staged rollout (canary → 10% → 50% → 100%) → verify health → GKG pulse update.
	•	Dependency: Requires G4 (skill-registry) and G8 (Cedar tests).

3.3.4 Composer migration completion [G11]
	•	Project: all_drupal_custom (multiple modules)
	•	Action: Publish remaining custom modules to GitLab group Composer registry. Update TESTING_DEMOS and WORKING_DEMOs composer.json to pull from registry. Verify composer update succeeds for all 5 OSSA WORKING_DEMOs.

3.4 Phase 4 — External & Hardening (Weeks 7+)
Goal: External publishing, security scanning, observability.

3.4.1 Security Scanner for skills [G14]
	•	Action: Supply-chain validation: dependency audit, code pattern scanning (no eval, no network calls without declaration), license compliance check. Integrate into fleet-ops GATE step.

3.4.2 Manifest ↔ export drift detection [G15]
	•	Action: CI job in gitlab_components that compares manifest declared capabilities against actual package.json exports. Fails MR if mismatch detected.

3.4.3 Token expiry alerting [G13]
	•	Action: n8n workflow: daily check of PAT expiry dates via GitLab API. Alert via webhook if any token expires within 7 days. Include n8n admin and MCP API tokens from 1Password vault.

3.4.4 AgentSkills.io publishing pipeline
	•	Action: Extend skill-registry with publish-to-external command. Format normalize → validate → push to AgentSkills.io registry API. Include CI job for automated publishing on tag.

3.4.5 Bloom/Research sync integration
	•	Action: Connect Zotero/Obsidian NAS research library with GKG for semantic skill discovery. Map research papers to relevant skills via @bluefly/agent-brain.

4.3 Required Design Decisions
	•	Skill-registry storage backend: GKG-native (Qdrant vectors + PostgreSQL metadata) vs. GitLab Package Registry metadata vs. hybrid. Recommendation: hybrid — GKG for semantic search, GitLab for versioned artifacts.
	•	Format normalizer scope: Standalone @bluefly/format-normalizer package vs. module within @bluefly/skill-registry. Recommendation: module within skill-registry (fewer packages, shared types).
	•	ECA action configuration model: Hard-coded flow names vs. dynamic discovery via workflow-engine /flows endpoint. Recommendation: dynamic discovery with cached manifest for offline fallback.
	•	Fleet skill deploy rollout strategy: Percentage-based (canary) vs. ring-based (dev → staging → prod fleet). Recommendation: ring-based for multi-environment, percentage within each ring.
	•	AgentSkills.io publishing trigger: Manual (BuildKit command) vs. automated (CI on tag to main). Recommendation: automated with manual gate (require explicit label 'publish-external' on MR).
	•	Marketplace user isolation model: Per-user GitLab project (current design) vs. per-user namespace in shared project. Recommendation: keep per-user project for strongest isolation; add cleanup automation for abandoned projects.
