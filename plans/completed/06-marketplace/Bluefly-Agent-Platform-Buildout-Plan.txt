1. Executive Summary
This plan defines the strategy and phased roadmap to transform the Bluefly.io Agent Platform marketplace from its current foundation (60 skills, 5 agents, 9 commands, 10 plugin directories) into a comprehensive, multi-IDE, OSSA-compliant agent ecosystem. The marketplace at plugins.blueflyagents.com will serve as the canonical distribution point for Claude Code, Cursor, Windsurf, and future IDE integrations.

Current State: Operational infrastructure with Cloudflare Tunnel → Oracle Cloud → Drupal backend. Core OSSA three-tier architecture established. marketplace.json plugins array is EMPTY — no plugins are formally registered despite 60+ skills existing on disk.
Target State: Fully populated marketplace with categorized, versioned plugins; cross-IDE packaging; automated CI/CD for plugin publishing; OSSA v0.4.6 compliance across all agents; A2A/MCP protocol bridges; Drupal CMS 2.0 backend integration; and enterprise-grade observability.
Timeline: 16 weeks across 4 phases — Foundation (Wk 1-4), Packaging (Wk 5-8), Integration (Wk 9-12), Scale (Wk 13-16).

3.2 Plugin Packaging Model
Each distributable unit becomes a Claude Code plugin with the following structure:

	•	.claude-plugin/plugin.json — Manifest (name, version, description, author, skills[], commands[], hooks[], mcpServers[])
	•	skills/ — SKILL.md files with frontmatter (name, description, disable-model-invocation, user-invocable)
	•	commands/ — Command definitions referenced by plugin.json
	•	hooks/ — hooks.json with PreToolUse/PostToolUse/SessionStart/Stop events
	•	.mcp.json — MCP server configuration for tool integration
	•	README.md — Plugin documentation and usage instructions

3.3 Marketplace Registry Schema
The marketplace.json plugins[] array will be populated with entries following the Claude Code marketplace schema:

	•	name — Plugin identifier (e.g., 'bluefly-agent-platform')
	•	source — Object with type ('relative', 'github', 'npm', 'url') and path/url
	•	description — Human-readable summary for IDE search
	•	version — Semver string
	•	author — Publisher identity (bluefly.io)
	•	category — One of: architecture, security, tdd, cost, git, ossa, openapi, drupal
	•	keywords — Search tags for discoverability

5.2 A2A Protocol Bridge
Google's Agent-to-Agent protocol enables peer-to-peer agent communication. An A2A bridge in Tier 3 (Go runtime) will expose OSSA agents as A2A-compatible endpoints, enabling interoperability with Google's agent ecosystem.

	•	Agent Card Mapping: OSSA manifest → A2A Agent Card (name, description, capabilities, endpoint)
	•	Task Protocol: A2A tasks mapped to OSSA workflow invocations via agent-mesh
	•	Discovery: A2A /.well-known/agent.json served from Drupal backend for each registered agent
	•	Authentication: OAuth 2.0 bearer tokens with Tailscale mesh as transport security layer

5.3 ACP (Agent Communication Protocol) Alignment
ACP from IBM provides structured agent-to-agent messaging. The compliance-engine will validate ACP message formats when agents operate in hybrid environments. This is a Phase 3 integration — monitor spec maturity before committing resources.

6. CI/CD & Publishing Pipeline
6.1 Plugin Publishing Flow
A GitLab CI pipeline in gitlab_components will automate the full plugin lifecycle from commit to marketplace publication.

	•	Stage 1 — Validate: Lint plugin.json, validate SKILL.md frontmatter, check OSSA manifest schema
	•	Stage 2 — Test: Run unit tests, integration tests against mock IDE environments
	•	Stage 3 — Package: Bundle plugin directory, generate checksums, create release archive
	•	Stage 4 — Publish: Update marketplace.json plugins[] array, push to static host, notify Drupal backend API
	•	Stage 5 — Verify: Smoke test plugin installation via Claude Code CLI, validate MCP server startup

6.2 Version Strategy
Version authority follows the existing OSSA model: HUMAN (milestone) → CI PIPELINE (semantic-release) → GIT TAG (output). AI agents remain READ-ONLY for all version information. Each plugin gets independent semver versioning, decoupled from the platform version.

11. Security & Compliance
11.1 Plugin Security Model
	•	All plugins signed with GPG keys from the bluefly.io keyring
	•	Checksums published alongside every release archive
	•	SAST scanning (GitLab SAST) on every MR, DAST on staging deployments
	•	Dependency scanning with GitLab dependency-scanning and license-scanning jobs
	•	MCP servers run with least-privilege: scoped API tokens, read-only filesystem by default
	•	Cedar policy evaluation for all tier_3_full operations (agent-docker, release-manager)
